## Ⅰ. 개요

### 1. 문서의 목적
본 문서는 "오픈소스 LLM·RAG 기반 PDF 양식 자동 작성 시스템(뚝딱)" 개발의 요구사항을 정의한 문서이다.
상기 서비스는 빈 PDF 양식과 사용자의 컨셉 입력만으로, 사내 문서를 근거로 양식의 각 항목을 자동으로 채운 완성 문서를 생성하는 것을 목표로 한다. 모든 처리를 사내(로컬)에서 수행해 외부 네트워크 호출이 발생하지 않는다.
이 요구사항 정의서는 PDF 양식 자동 작성 시스템 개발을 위한 설계문서를 작성하는 데 기초가 되며, 사용자 유스케이스를 기반으로 SW가 제공해야 할 기능 및 화면에서 필수적으로 구현되어야 할 요구사항에 대해 기술하고 정의한다.
본 문서는 가능한 구체적이며 간결하게 표현되어야 하고 추후 시험이 가능해야 하며, 본 문서를 사용하는 대상은 본 과제를 기획하는 기획자, SW를 개발하는 개발자 등이며, 본 과제의 요구사항 도출 및 개발 과정에서 본 문서를 활용할 수 있도록 한다.

### 2. 요구사항 및 문서의 범위
본 문서에서는 유스케이스 및 기능·테스트 요구사항의 기술을 그 범위로 한다.

## Ⅲ. 기능 요구사항

| ID | 요구사항명칭 | 설명 | 우선순위 |
|---|---|---|:---:|
| SFR-001 | 양식 업로드·폼필드 추출 | AcroForm 양식을 업로드해 필드명·라벨·타입·좌표·필수여부·최대길이를 추출한다 | 상 |
| SFR-002 | 라벨 자동 추론 | 위젯 주변 텍스트로 사람이 읽는 라벨을 추론한다 | 중 |
| SFR-003 | 컨셉 입력 | 작성할 문서의 컨셉·지시를 자연어로 입력한다 | 상 |
| SFR-004 | RAG 근거 검색 | 사내 지식 문서에서 top-k 근거 청크를 검색한다 | 상 |
| SFR-005 | 필드값 자동 생성 | 근거·컨셉 기반으로 각 필드 값을 생성한다(계산 포함) | 상 |
| SFR-006 | 근거/추론 판정 | 생성 값이 근거에 존재하는지 판정·표시한다 | 중 |
| SFR-007 | 미리보기·값 수정 | 근거/추론 구분으로 표시하고 값을 직접 수정한다 | 중 |
| SFR-008 | 좌표 한글 주입 | 추출 좌표에 한글 폰트로 값을 주입·평탄화한다 | 상 |
| SFR-009 | 멀티라인 문단 채움 | 서술형 박스는 줄바꿈·폰트 자동축소로 채운다 | 중 |
| SFR-010 | 완성 PDF 다운로드 | 채워진 PDF를 내려받는다 | 상 |
| SFR-011 | 지식 문서 인덱싱 | 사내 문서를 정제·청킹·임베딩해 저장한다 | 상 |
| SFR-012 | KPI 측정 | 정확도·근거율·시간단축·외부호출을 측정한다 | 하 |

## Ⅳ. 테스트 요구사항

| ID | 요구사항명칭 | 설명 |
|---|---|---|
| TER-001 | 폼필드 추출 정확도 확인 | AcroForm 양식에서 필드·타입·좌표가 정확히 추출되어야 한다 |
| TER-002 | 라벨 추론 정확도 확인 | 다열 양식에서도 라벨이 옆 칸과 섞이지 않고 추론되어야 한다 |
| TER-003 | 근거 검색 관련성 확인 | 질의에 관련 있는 사내 문서 청크가 상위로 검색되어야 한다 |
| TER-004 | 필드값 생성·계산 확인 | 생성 값이 근거에 부합하고 필드 간 계산(예: 공급가액=단가×수량)이 맞아야 한다 |
| TER-005 | 근거/추론 표시 확인 | 각 값의 근거/추론 구분이 올바르게 표시되어야 한다 |
| TER-006 | 한글 주입 깨짐 없음 확인 | 좌표 주입 시 한글이 깨지지 않고 렌더링되어야 한다 |
| TER-007 | 멀티라인 문단 렌더 확인 | 긴 서술형 문단도 사라지지 않고 박스 안에 채워져야 한다 |
| TER-008 | 완성 PDF 다운로드 확인 | 채워진 PDF가 정상적으로 생성·다운로드되어야 한다 |
| TER-009 | 외부 호출 0건 확인 | 실행 중 외부 도메인 네트워크 호출이 0건이어야 한다 |
| TER-010 | 채움 정확도 확인 | 폼필드 채움 정확도가 80% 이상이어야 한다 |
| TER-011 | 근거 일치율 확인 | 채운 값의 근거 일치율이 90% 이상이어야 한다 |
